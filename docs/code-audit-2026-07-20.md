# 아공맵 코드 감사 (2026-07-20)

감사 방식: 디자이너·프런트엔드·파이프라인 세 관점에서 각각 지적을 뽑고, 별도 검토자가 인용 검증 → 반박 시도 → 심각도 조정을 거쳤다. 23건 중 16건이 검토를 통과했고 7건은 기각됐다. **이 문서는 판단 자료이며 코드는 한 줄도 수정하지 않았다.**

---

## 1. 총평

구조는 건강하다. 3단 샤딩, fail-fast 배치, 외부 워치독, 디자인 감사 스크립트까지 이 규모 개인 프로젝트에서 보기 드물게 방어가 겹겹이 깔려 있고, 코드 안 주석이 "왜 이렇게 했는지"를 실제로 설명한다. 검토 과정에서 기각된 7건 중 상당수가 "이미 다른 층에서 막고 있어서" 기각됐다는 점이 그 방증이다.

문제는 거의 전부 **최근 변경의 뒷정리 누락**에 몰려 있다. 3단 샤딩(2026-07-19) 때 `sw.js`의 network-first 목록에 `data-trend.json`을 추가하는 것을 잊었고, 스코어 부호 표기를 뒤집으면서 생활권 허브의 리드 문장 한 줄만 옛 규약에 남았으며, 수도권 합계 행을 허브 표에 넣으면서 순위 분모를 맞추지 않았다. 세 건 모두 "설계가 틀렸다"가 아니라 "한 곳을 빠뜨렸다"에 해당한다. 디자인 시스템도 대부분 일관되게 적용됐고, 예외는 생성기(`make_zone_pages.py`)로 만들어지는 허브 한 장과 `.chg-up/.chg-dn` 한 줄뿐이다.

솔직히 덧붙이면, high 3건 중 2건은 사용자가 **틀린 숫자·틀린 부호를 보는** 문제라 우선순위가 명확하다. 반면 medium 이하는 대부분 "동작은 하는데 낭비이거나 접근이 막혀 있다" 수준이고, 몇 건은 제안된 수정을 그대로 적용하면 오히려 새 문제를 만든다 — 아래 각 항목에 그 경고를 함께 적었다.

## 2. 심각도별 발견

### HIGH

#### H-1. 생활권 허브의 부호 설명이 실제 표와 정반대다
- **어디서**: `zone/index.html:74` (생성원 `tools/make_zone_pages.py:509`)
- **관점**: 디자이너 / 심각도 유지
- **무엇이**: 리드 문장은 "양수는 공급이 모자란 곳, 음수는 남는 곳입니다"인데, 바로 아래 표 1위는 `−75,072 · 공급 절벽`(빨강)이고 최하위는 `+6,107 · 공급 과잉`(파랑)이다. 표 전체가 리드와 정확히 반대다.
- **왜 문제인가**: 화면 표기 규약은 `index.html:1913`의 `scFmt`가 정한 대로 **부족=음수**이고, 상세 38장은 `zone/*/index.html:100`에서 "음수(−)는 부족, 양수(+)는 초과입니다"로 올바르게 적는다. 허브 리드만 내부 데이터 규약(부족=양수)에 멈춰 있다. 허브는 `index.html:136` "전국 생활권 공급 순위"가 착지하는 첫 화면이라 오독의 진입점이다. 같은 화면에 교정 문구도 없다.
- **제안**: 리드를 상세와 같은 문장으로 통일. 생성물이므로 `make_zone_pages.py:509` 템플릿을 고쳐야 재생성 때 되살아나지 않는다. 포맷 충돌 없음(리터럴 `%` 없음).
- **남는 문제**: 지표 이름 자체가 "누적 순부족"이라 `−75,072`가 "순부족 마이너스"로 읽히는 마찰은 문장 수정만으로는 안 사라진다. 명명을 손볼지는 별도 판단.

#### H-2. 생활권 개수·순위가 화면마다 다르다 (37 vs 36, 순위는 1씩 어긋남)
- **어디서**: `index.html:136`, `zone/index.html:9·10·73·83`, `zone/*/index.html:93` (생성원 `tools/make_zone_pages.py:560·572·578` vs `:231-232`)
- **관점**: 디자이너 / **심각도 상향(medium → high)**
- **무엇이**: 홈·허브는 "37곳", 상세는 "36곳 중 N위". 원인은 허브가 개별 36 + 수도권 합계 1을 한 목록에 섞기 때문이다. 검토 과정에서 더 나쁜 사실이 드러났다 — **수도권 합계가 1위 자리를 차지해 그 아래 모든 행의 순위가 상세 페이지와 정확히 1씩 어긋난다**(서울권: 허브 2위 / 상세 1위, 대구권: 허브 3위 / 상세 2위 … 36장 전수 확인).
- **왜 문제인가**: 사용자가 보는 건 분모 차이가 아니라 "표에서 3위였는데 들어오니 2위라고 한다"이다. 홈은 이미 올바르게 처리한다 — `index.html:1984-1991`이 수도권을 1행으로 접고 `:2003`에서 "(16개 생활권)" 꼬리표를 단다. 처리되지 않은 유일한 화면이 허브다.
- **제안**: 허브를 고친다. 수도권 행을 순위 번호 밖 소계로 분리하고 문구를 "개별 생활권 36곳"으로 통일(title:9 / description:10 / lead:73 동시). `build_hub`의 `len(live)`를 `len(rows)` 기반 상수로 바꾸면 두 화면이 한 소스를 공유한다.
- **하지 말 것**: "상세 분모를 37로 맞춘다" 갈래는 채택하면 안 된다. 상위 단위와 구성원을 같은 모집단에 넣어 이중 계상을 정본으로 만든다.

#### H-3. `sw.js` network-first 목록에서 `data-trend.json`이 빠져 통계가 캐시에 고인다
- **어디서**: `sw.js:56-57` (대조 `index.html:442`)
- **관점**: 파이프라인(심각도 유지) + 프런트엔드가 같은 건을 독립 발견(그쪽은 **medium으로 하향**)
- **무엇이**: 조건문에 `/data.js`, `/app.css`, `/data-core.js`, `/data-rest.json`은 있는데 `/data-trend.json`만 없다. 따라서 `sw.js:85-96`의 cache-first 분기로 떨어져 `return hit || network;`로 캐시본이 즉시 반환된다.
- **왜 문제인가**: 재방문자는 통계 탭에서 지난주 주간 그래프를 본다. 더 나쁜 건 `split_data.py:73-74`가 **ADV 전체**를 trend에 싣고 `index.html:455`가 `Object.assign(ADV,d.ADV)`로 덮어쓴다는 점 — 통계 탭을 한 번 열면 지난주 ADV가 현재 세션의 전역 ADV를 되돌려, 홈 히어로 지도와 아공맵 스코어까지 지난주 숫자로 후퇴한다. `run_weekly_update.bat:111`의 `git add` 목록에 `sw.js`가 없어 배치는 VERSION을 절대 올리지 않는다.
- **심각도에 대한 솔직한 유보**: 두 검토자의 결론이 갈렸다. cache-first에도 백그라운드 갱신이 있으므로 지연은 "한 주"가 아니라 "한 방문"이고, 주 2회 이상 방문자는 거의 못 본다(→ medium 주장). 반면 금요일에만 들르는 사용자에게는 사실상 영구적으로 한 주 늦은 데이터가 보이고, 홈 지표까지 오염된다(→ high 주장). 수정이 한 줄이고 위험이 0이므로 **높은 쪽으로 잡고 처리하는 편이 합리적**이라고 본다.
- **제안**: 조건에 `|| url.pathname === '/data-trend.json'` 추가. `url.pathname.startsWith('/data-')`로 바꾸면 다음 샤딩 때 같은 누락이 재발하지 않는다(현재 `data-*` 파일 셋 다 주간 재생성물이라 안전). VERSION 상향은 필수가 아니라 선택.

## 4. 잘 되어 있어 건드리면 안 되는 것

- **`sw.js:49-55`의 판본 원자성 정책.** HTML·app.css·data-core를 network-first로 묶은 것은 사고가 아니라 명문화된 결정이다. `Promise.race` 타임아웃을 넣으면 각 문서가 독립적으로 승패를 갈라 판본 혼합이 기대값이 된다. 디자인 토큰을 통째로 교체한 직후라 비용이 특히 높다.
- **`run_weekly_update.bat`의 fail-fast 정책과 exit code 체계**(`:8-21`). 치명(10~17, 20)과 경고(`echo WARN:`)를 의식적으로 갈라 놨다. rc=13(share)을 경고로 낮추면 "이번 주"라 표시된 지난주 지도가 뉴스레터·인스타로 나간다.
- **`update_adv_data.py`의 옵션 키 설계.** ECOS·DATA_GO_KR 키 부재를 비치명으로 둔 것과 livezone 실패를 중단 사유로 승격하지 않은 것은 `:1113-1117` 가드(zone URL 404 방지)와 짝을 이룬다.
- **`split_data.py`의 CORE_STATS assert / CORE_ADV 무가드 비대칭.** 전세가율·주택멸실은 스코어 계산에 들어가는 값이라 누락 시 조용한 오정보가 되고, `ADV.weekly.sgg`는 장식 배경이다. 위험도가 다르니 가드가 달라야 한다.
- **홈의 수도권 롤업 처리**(`index.html:1984-1991`, `:2003`). 허브가 따라야 할 정답이다.
- **데이터 색 규약** — 빨강=상승/부족, 파랑=하락/과잉. M-3은 값의 통일이지 의미 변경이 아니다.
- **`advSel`의 select 상태 보존**(`index.html:1399-1403`). 재렌더가 사용자 선택을 덮지 않는다.
